Coding Style

Da Sie nun und in Zukunft mehr als nur wenige Zeilen Code schreiben und dazu übergehen, auch zusammenhängende Abläufe und Algorithmen zu programmieren, sollten Sie sich klar machen, dass und wie man dabei methodisch und strukturiert vorgeht.

Erst Beschreibung, dann Codierung

Ohne die Beschreibung dessen, was da in Java programmiert werden soll, können Sie keinen Code schreiben.

Wie sollen Sie Code schreiben, wenn Sie nicht wissen

  • Was der Code eigentlich leisten soll?
  • Wie der Code aufgebaut sein soll?

Daher ist es notwendig, VOR dem Schreiben des Java-Codes auf Deutsch (oder Englisch, ...) zu beschreiben, was codiert werden soll. Diese einfache Maßnahme erlaubt es Ihnen, die Aktivitäten LVÜAC, nämlich

  1. Nachdenken über das Problem (Lesen)
  2. Notieren von Anmerkungen (Verstehen)
  3. Nachdenken über die Lösung (Überlegen)
  4. Beschreibung der Lösung (Aufschreiben)
  5. Codieren der Lösung (Codieren)
    incl. Dokumentation von Implementierungsdetails

in mehreren Durchläufen in der richtigen Reihenfolge hintereinander ausführen zu können.

 

So brauchen Sie nicht auf allen diesen Baustellen (1 bis 5 oben) gleichzeitig Ihr Gehirn einzusetzen, was JEDEN Programmierer überfordert und was kein guter Programmierer tut.

Die richtige Vorgehensweise kostet Sie keine Zeit sondern spart im Gegenteil jede Menge Zeit, die Sie sonst mit Fehlersuche und Rätseln über Ihrem Code vergeuden.

Zudem werden Sie, wie Generationen vor Ihnen, feststellen:

Sie lesen Ihren Code sehr viel häufiger als dass Sie ihn schreiben.

Sie lesen Ihren Code nämlich, um sich zu orientieren, zu prüfen, was Sie noch codieren müssen, wo das hinkommt, warum zum Teufel dieser Fehler auftritt und wo überhaupt was passiert.

Das geht

  • schlecht, wenn da nackter Code steht, der erst wieder nachträglich enträtselt und verstanden werden muss
  • gut, wenn da steht, was der Code tun soll

Nutzen Sie daher die Möglichkeit von Java, zusätzlich zum Programmcode auch solchen Text in das Sourcefile zu schreiben, der den Code erläutert. (Der Compiler ignoriert diese Beschreibung).

Gehen Sie daher IMMER bei der Erstellung eines Programmes in der Reihenfolge LVÜAC vor!

Das mag Ihnen wie Zeitverschwendung – oder sogar uncool – vorkommen. Immerhin ersparen 30 Minuten Fehlersuche das Aufschreiben des Lösungswegs in 5 Minuten ;-) Dass dieses strukturierte Vorgehen aber im Gegenteil höchst professionell ist (und nicht: „vom Hirn in die Tastatur zu programmieren“ !), können Sie in allen Büchern von anerkannten Fachleuten nachlesen, z.B.:

Im Folgenden lernen Sie die verschiedenen Formen kennen, in denen Sie Beschreibungen, Anmerkungen oder Kommentare in Java-Code einbringen können.

Inline-Kommentare

Alles, was nach // in einer Zeile geschrieben wird, wird nicht vom Compiler als Java-Code ausgewertet. Das bedeutet, dass man nach // einen menschenlesbaren Kommentar anbringen kann. Ab dem Anfang der nächsten Zeile erwartet der Compiler wieder Java-Code.

Im folgenden Beispiel erklärt der Kommentar, wie die Anzahl zustande kommt.

int maxTeilnehmer = 25; // die Raumgröße begrenzt die Teilnehmer-Zahl

Im Beispiel oben wird der inline-Kommentar (nach//) dazu verwendet, im Code auf Deutsch

  • den (nächsten) Schritt zu beschreiben
  • oder eine Anweisung zu erklären

Dabei ist sinnvoll, dass Sie im Kommentar nicht noch einmal den Code wiederholen. Oben ist die Zuweisung von 25 zur Variable sinnvoll, weil inhaltlich, erläutert. Der folgende Kommentar wiederholt aber nur den Code und ist daher schlecht:

int maxTeilnehmer = 25; // maxTeilnehmer wird auf 25 gesetzt. 

Einen solchen wiederholenden Kommentar sollten Sie immer vermeiden, da

  • keine Zusatzinformation hinzukommt,
  • aber trotzdem mehr zu lesen und zu verstehen ist,
  • sie außerdem sowohl im Code als auch im Kommentar die 25 ändern müssten

Statt dessen sollte der Kommentar immer Zusatzinformationen, vor allem Anforderungen und Begründungen für den Code, bieten.

Namensgebung

Der C-:ompiler akzeptiert jeden Sourcecode solange der Syntax von Java gehorcht wird. Die Lesbarkeit des Textes für den Menschen ist ihm dabei nicht wichtig. Der folgende, offensichtlich schlecht lesbare, Code wird ohne Fehlermeldung übersetzt (wenn alle Variablen definiert und initialisiert sind):

while(a+hf<=wrg) if(hb*4+mpf<89%blb) System.out.println(" " + hb + " " + mpf-2);

Um den Sourcecode aber für sich selber und die Kolleg*Innen lesbar zu halten, sollte man sich an einige wenige Regeln halten. Nicht alle sind Ihnen jetzt schon verständlich, der Vollständigkeit halber werden aber schon die wichtigsten an dieser einen Stelle aufgeführt, damit Sie hier später nachschlagen können:

  • Namen beginnen (in Java) immer mit einem kleinen Buchstaben. Danach können kleine und große Buchstaben, Ziffern und _ verwendet werden
  • Namen sollten immer sinnvoll sein, da Sie sie sich so leichter merken und verwenden können
    • Namen sollten so lang sein, dass man sie verstehen kann
      • z.B: counter, zeile, spalte, spielfeld, summe
    • Namen sollten andererseits so kurz sein, dass der Code übersichtlich bleibt
      • eher nicht inHorizontalerRichtungAlleSpielsteineUmdrehen(),
      • sondern besser horizontalUmdrehen()
  • Zusammengesetzte Namen werden gebildet
    • entweder durch BinnenGroßBuchstaben (CamelCase notation)
    • oder durch binnen_unter_striche
  • Namen sollten nicht abgekürzt werden, da Sie sonst immer nachsehen müssen, wie die Abkürzung lautet.
    • D.h. z.B.: counter statt cntr
    • Ausnahmen sind Zählvariablen in Schleifen, diese können und dürfen i, j, x, y lauten

Wählen Sie nur englische oder nur deutsche Begriffe, mischen Sie sie nicht, im Zweifel verwenden Sie nur Englisch.

Formatierung

Eine übersichtliche Formatierung trägt wesentlich dazu bei, den Code schneller erfassen und verstehen zu können.

Abschnitte unterteilen

Verwenden Sie Leerzeilen zur Strukturierung von logischen Abschnitten. Erlaubt, aber offensichtlich sehr schlecht lesbar und unverständlich ist es, möglichst viel in eine Zeile zu quetschen:

public class Main{public static void main(String a[]){System.out.println("Hello World");}}

Erinnerung, wie Sie einen einzelnen Code-Abschnitt ausführen können:

  • Code-Abschnitt mit der Maus selektieren
  • Im Kontext-Menü (rechte Maustaste) "Führe nur selektierten Code im Code-Rahmen aus" auswählen

Eine sinnvolle Unterteilung in Code-Abschnitte wird außerdem durch die Verwendung der Aufgaben-Links unterstützt. Durch die generierten Kommentarzeilen mit der Überschrift und den Backlink sind sie sehr gut im Code-Rahmen erkennbar.  

Einrücken

Rücken Sie Ihren Code auf jeden Fall ein, wenn Sie eine Kontrollstruktur wie Verzweigung oder Schleife verwenden. Dadurch machen Sie klar, welcher Code zum Körper der Kontrollstruktur gehört.

Verwenden Sie Einrückungen (jeweils 1 Tabulatorschritt) durchgängig.

Anordnung von Block-Klammern { }

Für die Anordnung der Block-Klammern { und } gibt es mehrere Varianten, von denen Sie eine durchgängig (mindestens pro Termin) verwenden sollten:

Entweder verwenden Sie die etwas platzsparendere Variante, bei der ein Block in der gleichen Zeile wie z.B. eine Verzweigung oder Schleife beginnt:

// { direkt nach der if-Anweisung
if (a > b) {
    System.out.println("a ist größer als b");
}

oder Sie beginnen einen Block immer in einer neuen Zeile. So wird es sehr einfach, zu erkennen und zu prüfen, ob es eine { für jede } gibt:

// { in einer neuen Zeile
if (a > b)
{
    System.out.println("a ist größer als b");
}

Die beiden zusammengehörenden Klammern eines Blocks werden Ihnen übrigens auch in Entwicklungsumgebungen wie 5Code typischerweise markiert dargestellt, wenn Sie mit dem Cursor auf Klammern gehen, egal ob rund, geschweift oder eckig.

(Weitere Einrückungsstile finden sie u.a. in der Wikipedia)

Beschreibung von Prozeduren und Funktionen

Da Prozeduren und Funktionen einen zusammenhängenden Block von Anweisungen darstellen, die eine Aufgabe erfüllen, muss auch ihre Beschreibung ausführlicher sein.

Gute Namen

Ein guter Name für eine Prozedur oder Funktion ist, ebenso wie bei Namen von Variablen, sehr wichtig für das Verständnis und die Lesbarkeit von Code.

Allgemeine Grundsätze für gute Namen

  • Der Name ist kurz, aber lang genug, sodass man seine Bedeutung verstehen kann
    • um den Code lesbar und übersichtlich zu halten
    • z.B. groessterEintrag()
  • Aber nicht zu kurz
    • um den Code lesbar zu halten
    • um klar zu verstehen, was der Aufruf bewirkt
    • z.B. nicht: gE()
  • Abkürzungen sollte man vermeiden
    • um beim Schreiben des Aufrufs noch zu wissen, wie die Prozedur heißt
    • war das grEintr() oder grossEintrag() oder grEintrag() oder ...
  • Werden mehrere Worte zusammengesetzt, um den Namen der Prozedur zu bilden, gibt es im Wesentlichen zwei Konventionen:
    • CamelCase: DieWorteWerdenMitGrossenBinnenBuchstabenZusammengesetzt
    • Underscore: Die_Worte_werden_Unterstrichen_Zusammengesetzt
    • Halten Sie sich an eine Konvention in einem Kontext und mischen Sie sie nicht, um den Code lesbar zu halten
      • Im Java-Umfeld wird im Allgemeinen CamelCase bevorzugt.

Ein guter Name für eine Prozedur

  • Der Name einer Prozedur beschreibt, WAS passiert.
    • Der Name einer Prozedur entspricht damit einem Befehl.
    • z.B. druckerInitialisieren() oder initialisiereDrucker()
  • Der Code sagt dann, WIE es passiert.

 

Ein guter Name für eine Funktion

Für den Namen einer Funktion gilt, im Unterschied zu einer Prozedur:

  • Der Name beschreibt das Ergebnis der Funktion
  • damit das Ergebnis der Funktion weiter verwendet werden kann, ohne sprachlichen Schiffbruch zu erleiden
    • Beispiel: int g = groessterEintrag(); // sieht gut aus, ist verständlich
  • Er beschreibt aber (normalerweise) nicht, wie das Ergebnis zustandekommt
    • Also nicht: int g = berechneGroesstenEintrag(); // sieht blöd aus, ist umständlich

Javadoc

Damit man sich in jeder Beschreibung, auch von anderen Leuten, Unternehmen, Quellen, zurechtfindet, gibt es auch ein vorgegebenes Format, mit dem dieses beschrieben wird - diese Format heißt javadoc.

Mehr Informationen über das Format und die möglichen Tags finden Sie z.B. in der wikipedia: https://de.wikipedia.org/wiki/Javadoc. oder bei Oracle (dem aktuell für Java verantwortlichen Unternehmen): https://www.oracle.com/de/technical-resources/articles/java/javadoc-tool.html

In diesem Format wird alles beschrieben, WAS man wissen muss, um die Prozedur oder Funktion zu NUTZEN, d.h. sie korrekt aufzurufen.

  • Was soll diese Prozedur bzw. Funktion tun?
    • Diese Beschreibung findet in einem Blockkommentar statt, der mit /** beginnt und mit */ endet. Achten Sie auf die 2 * am Beginn.
  • Welche Rolle spielen die Parameter?
    • Jeder Parameter wird in 1 Zeile beschrieben nach einem speziellen Tag:
    • @param name beschreibung
  • Was wird als Ergebnis einer Funktion zurückgegeben?
    • Das Ergebnis wird in 1 Zeile beschrieben nach einem speziellen Tag:
    • @return beschreibung

Nicht beschrieben wird dagegen (meistens), WIE das genau passiert, wie die Funktion abläuft.

Ausnahmen davon sind z.B.:

  • eine bestimmte Reihenfolge von Aktionen ist wichtig
  • ein bestimmter Algorithmus soll verwendet werden

Wie was warum genau in der Funktion passiert, wird im Körper der Funktion beschrieben, meistens als Zeilenkommentare.

 

Eine gute javadoc-Beschreibung für eine Prozedur

Im javadoc-Kommentarblock /** ... */ bei der Definition einer Prozedur beschreiben Sie was man wissen muss, um die Prozedur verwenden zu können:

  • was in der Prozedur passiert
  • kurz, aber nicht zu kurz
  • ggf. notwendige Randbedingungen
  • welche Rolle die Parameter haben, welche Werte sie annehmen dürfen, etc

Beispiel:

/**
* Diese Prozedur initialisiert einen Drucker,
* der über ein Netzwerk angesprochen wird.
*
* Dabei wird
* - Die Netzwerkadresse eingerichtet
* - Der Tonerstand abgefragt
* - Der Papiervorrat abgefragt
*
* Wenn beim Bearbeiten dieser Ressourcen ein Problem auftritt,
* z.B. zuwenig Papier oder Toner, oder eine fehlende Netzwerkverbindung
* oder ein falscher Wert für den nutzerLevel ist angegeben,
* dann wird nichts gedruckt
*      und es wird ein Eintrag im zentralen Logfile vorgenommen.
*
* @param nutzerLevel kann 0=Allgemein oder 1=Spezial sein
*/
public void druckerInitialisieren(int nutzerLevel) {
// ...
}

 

Eine gute javadoc-Beschreibung für eine Funktion

Im javadoc-Kommentarblock /** ... */ beschreiben Sie - ähnlich wie bei der Definition einer Prozedur - was man wissen muss, um die Funktion verwenden zu können:

Ähnlich wie bei der Definition einer Prozedur, aber doch etwas anders, wird beschrieben:

  • was von der Funktion berechnet und zurückgegeben wird
  • d.h. das Ergebnis der Funktion
  • kurz, aber nicht zu kurz
  • ggf. notwendige Randbedingungen
  • welche Rolle die Parameter haben, welche Werte sie annehmen dürfen, etc

Und nun noch zusätzlich, speziell für eine Funktion:

  • wie genau das Ergebnis aussieht, welche Werte es annehmen kann, etc.

Beispiel:

/**
* Diese Funktion berechnet den größten Eintrag in einer
* Tabelle von Umsätzen.
* Die Tabelle muss nicht sortiert oder geordnet sein.
* Die Tabelle kann auch leer sein.
*
* @param umsatzTabelle beinhaltet alle relevanten Umsätze
* @return der größte Eintrag in der umsatzTabelle, -1.0f wenn sie leer ist
*/
float groessterEintrag(float [] umsatzTabelle) {
// ...
}
