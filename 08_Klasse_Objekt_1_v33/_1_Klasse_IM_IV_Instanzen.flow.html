Spiel FiveCrush entwickeln

Schrittweise über die nächsten Termine entwickeln

In diesem und den folgenden Praktikums-Terminen bis zum Ende des Semesters entwickeln Sie Schritt für Schritt ein Spiel.

Dabei setzen Sie nach und nach immer neue Eigenschaften der Objektorientierten Programmierung ein. 

Außerdem programmieren Sie das Spiel so, dass es zuerst in Textform, und nach und nach attraktiver mit grafischen Elementen auf der Grafik-Konsole dargestellt wird. 

In diesem ersten Termin definieren Sie die Klasse, mit der eine Spiel-Karte im Programm modelliert wird. Sie definieren für diese Klasse Instanz-Variablen und einige einfache Instanz-Methoden.  

Diese Klasse erweitern Sie in den folgenden Terminen schrittweise.

 -> Ihre Klasse, die IV und IM müssen in den jeweils folgenden Terminen funktionieren, da Sie darauf aufbauen werden!

 

Spielregeln

Das Spiel FiveCrush ähnelt Candy Crush und anderen Match3-Spielen:


(Bildquelle: Artstation.com)

Auf einem Spielfeld werden (bunte) Karten zufällig ausgelegt.
Bei einem Spielzug werden Karten ausgewählt und auf dem Spielfeld miteinander getauscht.
Wenn auf dem Spielfeld mindestens drei nebeneinander liegende Karten zueinander passen, werden Sie vom Spielfeld genommen. Dann werden neue zufällige Karten an ihrer Stelle ausgelegt.
Bei jedem Zug werden Punkte ermittelt und gesammelt. 

 

Die Klasse Karte

Bezug zur Objektorientierten Programmierung

Eine Karte ist ein "Ding", ein "Objekt", das auf einem Spielfeld liegt. Es hat folgende Eigenschaften:

Eine Karte kennt

  • ihre Position auf dem Spielfeld,
  • und die Art und Weise, wie sie dargestellt wird.

Eine Karte kann man

  • erzeugen und darstellen,
  • man kann ihre Eigenschaften verändern,
  • man kann sie mit einer anderen tauschen,
  • man kann sie anklicken
  • und vom Spielfeld nehmen,
  • und noch einiges mehr, was Sie später hinzufügen. 

Eine Karte kann man also als Programm-Objekt mit diesen Eigenschaften auffassen. Sie beginnen nun also (endlich) mit Objektorientierter Programmierung :-).

Daher definieren Sie nun - über mehrere Termine hinweg - eine Klasse Karte mit Instanzvariablen und Instanzmethoden, die diese Eigenschaften für alle Objekte der Klasse Karte abbilden. 

Variablen, Prozeduren und Funktionen haben Sie bereits in einer Klasse definiert und getestet - ohne die Rolle der Klasse dabei zu kennen.
Jetzt ist neu, dass Sie so die Eigenschaften von Objekten definieren: In Form von Instanz-Variablen und Instanz-Methoden. 

Der Coding Style wurde entsprechend um das Kapitel 4 - Beschreibung von Klassen ergänzt.

Um zu erfahren, was man mit einer Karte tun kann, zieht man ihre Javadoc-Beschreibung zu Rate. Daher schreiben Sie, wie bisher, für jede Eigenschaft eine kurze Beschreibung.

-> Ihre Klasse, die IV und IM müssen in den jeweils folgenden Terminen funktionieren, da Sie darauf aufbauen werden!

 

Konkrete Eigenschaften einer Karte

Eine Karte (d.h. ein Objekt der Klasse Karte) muss sich unter anderem diese Eigenschaften merken, die mit der grafischen Darstellung zusammenhängen:

  • Position auf dem Spielfeld: Zeile und Spalte als ganze Zahlen
    • Zeile = 0 ist die oberste Zeile
    • Spalte = 0 ist die Spalte am linken Rand
    • Der Wert für Zeile und Spalte muss größer oder gleich 0 sein
  • Farbe und Text für die Darstellung
    • Beide werden als String definiert, um die Karte später auf der Grafik-Konsole darstellen zu können.
    • Der Text wird auf der Karte angezeigt.
      • Der Text soll maximal 3 Zeichen lang sein.
    • Die Farbe ist die Farbe des Hintergrunds der Karte.
      • Da Farben immer mit der Grafik-Konsole zu tun haben, gibt es vordefinierte Ausdrücke für die erlaubten Farben:   
        Grafik.BLACK, Grafik.WHITE, Grafik.GREY, Grafik.RED, Grafik.BLUE, Grafik.GREEN. Grafik.YELLOW, Grafik.MAGENTA, Grafik.CYAN.
        Diese Ausdrücke haben alle den Datentyp String. 

Diese Eigenschaften erweitern Sie in den nächsten Terminen, z.B. für die Darstellung auf der Grafik-Konsole. Das Feature Grafik-Konsole muss aber auch schon in diesem Termin aktiviert sein, damit die oben aufgeführten Farb-Ausdrücke gültig sind. 

Ein paar Hinweise zu Ausdrücken wie Grafik.RED

Einfache Instanz-Methoden definieren

Nun definieren Sie einige einfache Aktionen, die mit (oder "für" oder "auf") einer Karte ausgeführt werden können als Instanz-Methoden (IM).

-> Der Test dieser Methoden folgt im folgenden Abschnitt 2.6.

In den Instanz-Methoden der Klasse Karte sollen die Instanz-Variablen der Karte geändert werden, aber nicht auf einen beliebigen Wert: Um die angegebenen Randbedingungen für jede dieser Variablen einzuhalten, werden die übergebenen Parameter vorher geprüft und ggf. passendere Werte in die IV gesetzt. Instanz-Methoden für die Spiel-Logik 

Nachdem die Basis-Methoden für eine Karte definiert sind, implementieren Sie nun einige Methoden, die im Verlauf des Spiels eingesetzt werden.  

In diesem Methoden spielt nicht nur das Ziel-Objekt eine Rolle, sondern auch eine zusätzliche Karte, die dafür als Parameter übergeben werden muss.

 

Mit einer anderen Karte tauschen.

Im Verlauf des Spiel werden Karten auf dem Spielfeld miteinander getauscht. Das bedeutet, dass danach die Positionen beider Karten-Objekte vertauscht sind. Die anderen Eigenschaften der Karte bleiben (natürlich) erhalten. 


In diesem Beispiel werden die gelbe und die grüne Karte getauscht: sie wechseln die Position, ihre Farbe (und ihr Text) bleibt aber erhalten. 

Für die Umsetzung ist wichtig: Sie greifen NUR auf die IV eines Objekts zu, das eine Instanz der gleichen Klasse ist, wie das Ziel-Objekts. Sie sollten aus den bekannten Gründen nicht auf die IV von Instanzen von anderen Klassen zugreifen: das verstößt gegen das Konzept der Kapselung!

In diesem Fall sind beide beteiligten Objekte von der gleichen Klasse. 

Die Methode wird für eine Karten-Instanz (d.h. ein Karten-Objekt) aufgerufen

Auch für das Setzen der neuen Position jeder Karte wird die oben dafür definierte Instanz-Methode für diese Karte aufgerufen.

Da nun für eine neue Karte auch die ID ermittelt und gespeichert wird, kann auch geprüft werden, ob das Objekt eine bestimmte ID hat.

Troubleshooting bei der Erstellung der HTML-Dokumentation

Beim Erstellen der HTML-Dokumentation können ein paar Probleme auftreten - genauso, wie beim Compilieren.

Kommentar im Javadoc-Format kann man mit HTML-Tags formatieren. Das führt dummerweise dazu, dass die Zeichen > und < im Text zu Fehlern führen. Vermeiden Sie also diese Zeichen und ersetzten Sie sie z..B. durch die ausgeschriebenen Worte größer oder kleiner.
 
Um Javadoc-Dokumentation erzeugen zu können, muss das Java-File syntaktisch korrekt sein.
 
Manchmal werden Warnings in der Konsole in rot angezeigt. Diese führen aber nicht zum Abbruch der Erzeugung der Dokumentation und müssen daher nicht behoben werden.

Javadoc als einzige Informationsquelle

Auf diese Art und Weise werden alle Klassen beschrieben,

  • die hunderte von Klassen, die mit dem installierten JDK mitgeliefert werden, z.B. String
  • die Klassen, die Sie selber schreiben 

Diese Form der Dokumenation ist fast immer die einzige Informations-Quelle, wenn Sie Klassen nutzen möchten. Alle Klassen, Variablen und Methoden müssen im Javadoc-Dokument so beschrieben werden, dass man sie nutzen kann - ohne den Quellcode zu sehen oder zu kennen.

Quelle von anderen als den eigenen Java-Klassen bekommt man fast nie zu sehen.   
 

Die Methoden, die für die Klasse String bereitgestellt werden, finden Sie in der Javadoc-Beschreibung von String in der "API-Dokumentation" von Java.
