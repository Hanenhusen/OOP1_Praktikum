Vorbereitung und gameloop

Alle Tests schreiben Sie in die Methode main() der Klasse FiveCrush.

Diese Methode main() bereiten Sie für die Tests in diesem Termin wie folgt vor: 

  • Löschen Sie den bisherigen Inhalt von main()
  • In main() soll nun folgendes passieren
    • Grafikkonsole initialisieren
    • Spielfeld erstellen und zufällig füllen
      • Spielfeld zur Kontrolle als Text auf der Konsole ausgeben.
    • Ende Button erstellen.
    • Variablen für Ergebnisse der folgenden Aktionen erstellen
      • Für die Eingabe (Klickantwort) 
      • Für eine gefundene Karte
      • Für die Wiederholung der gameloop
    • gameloop: Solange weitergespielt werden soll, wiederholen von:
      • Eingabe von der Konsole (Klick) einlesen und auswerten:
        • Wenn auf den Ende-Button geklickt wurde, soll die Wiederholung der gameloop gestoppt werden.  
        • Sonst wird die Eingabe ausgewertet, was in den folgenden Aufgaben entwickelt wird.
    • Nach Beendigung der gameloop geben sie "Spiel beendet" auf die Textkonsole aus.

Im Folgenden erweitern Sie die main()-Methode so, dass sie die neuen Klassen und Methoden dieses Termins testen können.

Top-Down-Entwicklung - Objektorientiert

Sie entwickeln nun Top-Down die Anwendung weiter. D.h. Sie beginnen bei einzelnen Aspekten einer Funktionalität (oben) und überlegen, wie diese umgesetzt werden können. Dabei ergeben sich weitere Anforderungen, die ihrerseits umgesetzt werden, wie ausführlich bereits in der Vorlesung dargelegt. 

Objektorientiert gehen Sie dabei vor, weil für jede Umsetzung (Variable, Methode) überlegt wird, in welche Klasse diese gehört. Dabei ist auch wichtig, welche Klasse auf welche Eigenschaften zugreifen darf.

Außerdem folgen Sie dem OO-Weg "Beauftragen statt Zugreifen".

 

Spezielle Karten sollen auf Klick speziell reagieren

Weiter im Spiel: Das Klicken auf eine Spiel-Karte (bzw. auf den damit verbundenen Button in der Grafik-Konsole) soll eine spezielle Aktion für den Typ dieser Karte auslösen.

Damit für Objekte verschiedener Klassen verschiedene Dinge passieren, definiert man in jeder Subklasse von Karte eine Methode mit der gleichen Signatur (Name und Parameterliste) auf eine andere Art und Weise. Damit wird die Methode in der Subklasse überschrieben (redefiniert).

Sie üben daher in der folgenden Aufgabe

  • sowohl das Redefinieren/Überschreiben einer Methode der Superklasse,
  • als auch das Aufrufen einer Methode der Superklasse.

 

Horizontale Karten tauschen mit dem Nachbarn

Als "horizontale" Karten-Typen werden hier die Klassen RKarte und LKarte bezeichnet, da sie auf Nachbarn in horizontaler Richtung einwirken.

Reaktion von RKarte

Zuerst redefinieren Sie die Klick-Reaktionsmethode (z.B. klickAuf() ) für die Klasse RKarte. In der Reaktion sollen folgende Schritte passieren:

  • Die RKarte soll auf die Text-Konsole ausgegeben werden.
  • Die RKarte tauscht mit dem rechten Nachbarn die Position.
  • Der Punktestand wird um 1 Punkt erhöht

Dabei verwenden Sie für jeden Schritt eine Methode, die Sie bereits in einem vorangegangenen Termin geschrieben hatten. Der Body der Methode sollte also sehr kurz und einfach aussehen. 

Beim Implementieren werden Sie feststellen, dass Sie der Compiler nicht auf das (geerbte) Spielfeld der RKarte zugreifen lässt. Warum ist das so?

Da die Subklassen von Karte eng mit der Karte verwandt sind, und sie alle gemeinsam erstellt werden, ist es daher zulässig, die Sichtbarkeit dieser Instanz-Variablen so zu ändern, dass sie auch in Subklassen zugreifbar ist. 

Ändern Sie die Sichtbarkeit dieser IV entsprechend. Alle anderen IV sollen weiter private bleiben. 

Redefinieren/Überschreiben Sie nun die Klick-Reaktions-Methode (klickAuf() ) in der Klasse RKarte.

Für jeden o.a. Teil der Reaktion verwenden Sie bereits erstellte Methoden.
 
 
Wichtig: Geben Sie die Karte hier nicht (!) explizit mit System.out.println() auf die Text-Konsole aus!

Geklickte Karte finden und ihre Reaktion testen

Zum Testen der Reaktionen hatten Sie bereits ein Spielfeld aufgebaut und in einer gameloop als Reaktionen die IDs von geklickten Karten abgerufen und als Text ausgegeben.

In der gameloop wird nun die Aktion definiert, die aufgerufen wird, wenn der eingelesene Reaktionstext nicht der des Ende-Buttons war (siehe Beschreibung zu Beginn):

  • Fragen Sie das Spielfeld nach der Karte, die diesen Reaktionstext zurückgegeben hat.
    • Wenn die Karte gefunden wurde, wird die Klick-Reaktionsmethode für diese Karte aufgerufen,
    • andernfalls wird die gameloop und damit main() beendet.


Ergänzen sie nun die gameloop um diese Funktionalität und testen Sie so die Reaktionen der angeklickten Karten. Die hierfür notwendigen Methoden hatten Sie bereits geschrieben, verwenden Sie sie nun in der gameloop.

Machen Sie sich klar: 

  • Zur Laufzeit wird immer die Klick-Reaktionsmethode (z.B. klickAuf() ) der Klasse des aktuellen gefundenen Objekts aufgerufen. Das ist eine Instanz einer der Klassen
    L, R, U oder OKarte.
  • Ohne, dass Sie den Code zum Aufruf der Klick-Reaktionsmethode  geändert haben, wird unterschiedlicher Code ausgeführt - je nachdem, welcher Klasse die geklickte Karte angehört. 
  • Die Klick-Reaktionsmethoden haben alle die gleiche Signatur, sie führen aber unterschiedliche Aktionen durch.

Nachbarn mit Wrapraound implementieren

Modifizieren Sie die Methode, die den Nachbarn einer Karte ermittelt (z.B. nachbarVon()) so, dass ein "Wraparound" entsteht. D.h. der Nachbar einer Karte am linken Rand ist die Karte der gleichen Zeile am rechten Rand und umgekehrt. Gleiches gilt für den oberen und unteren Rand. 

In der Skizze ist dann die Karte "untenlinks" der linke Nachbar von der Karte "untere", und die Karte "untere" ist der rechte Nachbar von der Karte "untenlinks"

Damit können z.B. RKarten und LKarten "im Kreis" geklickt werden und bleiben nicht am rechten Rand stehen. So kann man während des Spiels mehr und andere Züge machen. 

Überlegen Sie: auch hier reicht es wieder, eine einzige Methode zu modifizieren, um den gewünschten Effekt in allen Karten-Klassen nutzen zu können.
